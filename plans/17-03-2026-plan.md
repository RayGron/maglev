# Maglev CLI Plan

## Goal

Develop `maglev` into a standalone C++ CLI agent with a Claude Code-like terminal workflow.

## Product Plan

1. Convert the CLI from single-run execution into a persistent chat session.
2. Introduce session state for plan, edits, approvals, commit state, push state, and deploy state.
3. Add terminal commands such as `/task`, `/status`, `/plan`, `/apply`, `/checks`, `/commit`, `/push`, `/deploy`, and `/exit`.
4. Implement selective edit review so the user can apply only approved files.
5. Add commit message review before running `git commit`.
6. Add deploy preview and explicit approval before SSH/deploy execution.
7. Improve terminal UX with clearer sections, step statuses, and readable failures.
8. Add error recovery so failed steps return the user to the session instead of terminating the process.
9. Add local transcript and audit logging for prompts, approvals, executed commands, commit hashes, and deploy results.
10. Finalize release ergonomics with stable CLI flags for task execution and automation.

## Backlog

### Phase 1: Session-Based Chat

1. Refactor [`main.cpp`](../cpp/src/main.cpp)
   Split one-shot startup from a reusable interactive session loop.
2. Expand [`chat.cpp`](../cpp/src/chat.cpp)
   Add a persistent REPL-style flow instead of a single prompt followed by execution.
3. Introduce session state storage
   Keep session state in `cpp/src/session.cpp` and `cpp/include/maglev/session.h` to track the active task, run status, pending approvals, and results.

### Phase 2: Structured Commands

1. Extend [`chat.cpp`](../cpp/src/chat.cpp)
   Parse slash commands and route them to runtime actions.
2. Update [`contracts.h`](../cpp/include/maglev/contracts.h)
   Add any state or payload types needed for explicit local command handling.
3. Update [`execution.cpp`](../cpp/src/execution.cpp)
   Split apply/check/commit/push/deploy into steps that can be invoked independently from the chat loop.

### Phase 3: Edit Review

1. Add file selection support in [`chat.cpp`](../cpp/src/chat.cpp)
   Let the user inspect proposed edits and choose a subset of files to apply.
2. Update [`execution.cpp`](../cpp/src/execution.cpp)
   Accept selected edits instead of always applying the full proposal set.
3. Improve output formatting
   Show file paths, summaries, and explicit `applied/skipped` results.

### Phase 4: Commit Review

1. Extend [`gateway.cpp`](../cpp/src/gateway.cpp)
   Keep commit message generation separate from commit execution.
2. Update [`chat.cpp`](../cpp/src/chat.cpp)
   Show generated commit title/body and let the user edit them before approval.
3. Update [`execution.cpp`](../cpp/src/execution.cpp)
   Commit with the reviewed message rather than the raw generated proposal.

### Phase 5: Deploy Review

1. Expand [`execution.cpp`](../cpp/src/execution.cpp)
   Produce a structured deploy preview before running SSH.
2. Extend [`chat.cpp`](../cpp/src/chat.cpp)
   Show `host`, `branch`, and remote command before approval.
3. Keep backend defaults in [`model-endpoints.json`](../config/model-endpoints.json)
   Continue to centralize endpoint and runtime profile configuration.

### Phase 6: Reliability and Logs

1. Add transcript persistence
   Create a new module such as `cpp/src/logging.cpp` for local session/audit logs.
2. Improve failure handling in [`chat.cpp`](../cpp/src/chat.cpp)
   Keep the session alive when checks, commit, push, or deploy fail.
3. Add configurable output and runtime flags in [`main.cpp`](../cpp/src/main.cpp)
   Support `--task`, `--auto-approve`, `--config`, `--backend`, and `--model`.

## Order of Execution

1. Session state and chat loop
2. Slash commands
3. Selective edit review
4. Commit review
5. Deploy review
6. Audit log and recovery
